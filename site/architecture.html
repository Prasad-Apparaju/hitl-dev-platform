What HITL is made of

Five kinds of building blocks compose into every workflow on this site. None of them is exotic on its own; the system property comes from how they interlock — instructions that anyone can read, enforcement that nobody can skip, and memory that outlives every session.

Who invokes what

Three different hands operate the system, and knowing which is which explains most of the architecture.

YOU · DELIBERATE

Humans invoke skills as slash commands (/hitl:ops-deploy), hold the approval gates, and make every promote, hold, and rollback call. Some catalog steps are marked manual: they are yours entirely.

CLAUDE · INSTRUCTED

Claude executes skill instructions, routes a change to the right next skill, drafts every document and test, and runs the reviewer agents. Steps marked guided mean Claude follows a reference doc without a dedicated skill.

THE HARNESS · AUTOMATIC

Hooks fire on events — session start, every prompt, before and after every file edit — not on anyone's judgment. Neither the human nor the AI can forget to run them. This is the lane that makes the process enforced.

How they compose into a workflow step

Every step in every workflow executes the same way. This is the whole trick — one repeating composition, not fifty special cases.

1 · DEFINED The CATALOG names the step: its stable key, its phase, the command that executes it, and the role that owns it — e.g. deploy · Ship · ops-deploy · Ops.
↓
2 · INVOKED A human types the slash command, or Claude routes to it as the next step of the active change. Steps marked manual are done by a person; guided steps follow a reference doc.
↓
3 · EXECUTED Claude runs the SKILL instructions — reading the durable memory for context, drafting from TEMPLATES, and calling AGENTS where an independent judgment is required.
↓
4 · GUARDED HOOKS run around every action: no edits without an active change, no cross-domain leaks, hard gates where the stakes demand it. Skills also carry refusal rules — conditions under which they stop and say why.
↓
5 · RECORDED The step's outputs land in DURABLE MEMORY — docs updated, registries appended, the change file's breadcrumb advanced — so the next step, and the next change, start from recorded truth.
One real step, traced end to end

What actually happens when someone runs /hitl:ops-deploy production

  1. The CATALOG defines deploy as a Ship-phase step owned by Ops, executed by the ops-deploy command.
  2. The ops engineer invokes the SKILL. Its first moves are refusal rules: no verified build, no deploy; and for production, it must run the readiness pre-flight.
  3. The pre-flight is a HOOK SCRIPT — check-platform-ready.sh — that validates the readiness register (built from a TEMPLATE) deterministically. Unknown status? Incomplete waiver? Truncated file? It exits blocked. The skill is instructed to stop and never offer a bypass.
  4. If the gate passes, the skill reads the approved rollout plan from DURABLE MEMORY — a plan whose canary criteria came from the incident registry — and presents the exact deploy command to the human before running anything.
  5. The human says go. Claude executes, then hands off to canary monitoring, where Claude reads dashboards against the go/no-go numbers and recommends — and the human decides.
  6. The outcome is RECORDED: the change file advances, the deployment is traceable back through rollout plan, reviews, tests, design, and issue.

The design rule underneath all of it: anything that must never be skipped is a hook, not an instruction. Anything that requires judgment is a skill with a human gate, not automation. Anything worth remembering is written to durable memory, not left in a conversation. That division of labor — deterministic enforcement, instructed intelligence, recorded truth — is what makes an AI-native process trustworthy at scale.
