Phase 2 · in parallel

Two lanes, one register

Changes can be made the moment onboarding ends. They can be delivered to customers only when the platform readiness register is green — the lanes run side by side.

Lane A — start making changes now

available immediately after step 11

  1. Pick the first issue (from step 10, or shape one with /hitl:pm-design-feature)
  2. /hitl:dev-start-change — classifies the workflow, seeds the change file, creates the branch
  3. Run the 31-step delivery loop (below)
  4. First touch of an undocumented component? /hitl:dev-generate-docs for it, then resume — this friction fades as docs accumulate

Lane B — stand up the platform NEW 1.1.0

replaces the hand-written roadmap issue

  1. /hitl:ops-plan-platform derive — register built from steps 5–6 verdicts; mark what already works verified, with evidence
  2. /hitl:ops-plan-platform roadmap — remaining gaps become GitHub issues across the nine pillars: test suites, E2E, traceability, reproducible build, deploy playbook, CD, observability, canary, security posture
  3. Each roadmap issue is an ordinary HITL change — it runs through Lane A's loop
  4. /hitl:ops-plan-platform verify-ready — four pillars checked → delivery_ready: true
GATE
Production deploy gate — until the register says delivery_ready: true, every Tier 2+ production deploy is blocked (fails closed), unless each open gap carries a recorded waiver: owner, revisit date, tier limit, reason. Staging and canary deploys are never blocked — Lane B needs them. The statusline shows the open-gap count until the project is delivery-ready.
Phase 3 · steady state

The enhancement loop

Every enhancement is one GitHub issue through /hitl:dev-start-change, which routes it onto the delivery spine:

Issue→ Design→ TA gate→ Build (TDD)→ Verify→ Assess→ TA gate→ Ship→ Post-ship

Design: impact analysis, docs, test plan, decision packet · Build: RED → QA test review → GREEN → refactor → conventions · Verify: two review rounds + architect review + QA verification · Ship: build → deploy → canary → monitor — where the production gate applies until delivery-ready.
