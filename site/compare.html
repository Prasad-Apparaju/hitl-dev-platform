Brownfield with HITL: 1.x vs 2.x

The journey is the same shape on both lines — install, 11-step onboarding, parallel platform standup, gated delivery. What differs is the workflow model underneath the enhancement loop.

Where they stand today

1.x — v1.1.0 LIVE TO CUSTOMERS

Marketplace pin serves 481ac0f. Numbered 31-step workflow + the full platform-bootstrap feature (register, roadmap, hard production gate), validated across six adversarial review rounds.

2.x — 2.1.0 BUILT, NOT PUBLISHED

Everything 1.1.0 has, on the redesigned workflow model. Publishing is one pin flip after a first 2.1.0 build — the existing 2.0.x build predates the platform feature and the build-pipeline fix.

Identical on both lines

The bones don't change

  • Install + hook wiring (8 hooks, settings, ADR stubs)
  • The 11-step brownfield onboarding skeleton
  • Steps 5–6 persist pipeline/observability verdicts to the readiness register
  • The two-lane fork: enhancements start immediately; platform standup runs in parallel
  • /hitl:ops-plan-platform — derive / roadmap / status / verify-ready
  • The production deploy gate — byte-identical check-platform-ready.sh, fails closed
  • Registries, decision packets, TA gates, TDD-first build discipline
  • Docs-on-first-touch for undocumented components
What the 2.x model changes
1.x (running today)2.x (next)
Step identity Global numbers — "Step 14 of 31"; renumbering breaks references Stable key + name + phase; numbers are display-only, derived from one catalog
Step plan One fixed 31-step sequence for every change Plan is an output of impact analysis: 6 profiles (Feature, Enhancement, Fix, Tech Change, Upgrade, Security) + 5 tags propose the conditional steps and required evidence, over a non-skippable floor
Breadcrumb Windowed step trail with a global counter Phase ribbon — Requirements ✓ · Design ✓ · Build ◐ · … — names where you are, no counter
Onboarding step 8 Seeds test + incident registries Also initializes the PRD shell (personas + format) — the PM and QA skill families work from day one instead of failing on a missing PRD
Docs-only changes No special route — full workflow or nothing Own 6-step docs workflow (issue → scope → draft → domain-routed review → reconcile → merge); mixed docs+code stays on the spine
Stale change files A concluded change file can linger and satisfy the session gate status: merged counts as inactive — a finished change never gates the next one
Manifest drift Checker exists in the platform repo only Ships with the plugin, copied into the product repo at onboarding; scan roots derived from the manifest itself
Taxonomy 6 workflows (development, brownfield, migration, migration review, prd, platform); one fixed step sequence each 7 workflows (adds the docs route) / 6 profiles / 5 tags — one shared spine underneath
What upgrading means for a live project: the change-file schema is additive and /hitl:dev-update remaps the embedded workflow by stable key — in-flight changes keep their done/current step status across the model change. The platform readiness register carries over untouched (schema_version 1.0 on both lines). 2.x ships as a major version because the mental model is different, not because anything breaks.
