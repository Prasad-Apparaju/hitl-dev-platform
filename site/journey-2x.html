Phase 2 · in parallel · identical on both lines

Two lanes, one register

Changes can be made the moment onboarding ends; delivered to customers only when the readiness register is green. The register, the skill, and the gate are byte-identical across 1.x and 2.x.

Lane A — start making changes now

available immediately after step 11

  1. Pick the first issue, or shape one with /hitl:pm-design-feature
  2. /hitl:dev-start-change — classifies the change (development, docs-only, or an onboarding route), seeds the change file, creates the branch
  3. Run the delivery loop (below)
  4. Undocumented component on first touch? /hitl:dev-generate-docs first, then resume

Lane B — stand up the platform

generated, not hand-written

  1. /hitl:ops-plan-platform derive — register from steps 5–6 verdicts; mark what works verified, with evidence
  2. /hitl:ops-plan-platform roadmap — gaps become GitHub issues across the nine pillars
  3. Each roadmap issue is an ordinary HITL change through Lane A's loop
  4. /hitl:ops-plan-platform verify-ready → delivery_ready: true
GATE
Production deploy gate — Tier 2+ production deploys blocked (fails closed) until delivery_ready: true or every open gap carries a recorded waiver (owner, revisit, tier limit, reason). Staging and canary never blocked. Identical enforcement on both lines.
Phase 3 · steady state · the 2.x difference

The enhancement loop, numberless

The plan is an output of impact analysis

Steps have stable identities (key + name + phase), never global numbers. /hitl:dev-start-change classifies the change onto a profile, tags refine the required evidence, and impact analysis finalizes which conditional steps apply — so a bug fix and a security-sensitive feature get different plans from the same spine, with a non-skippable floor.

Phase-ribbon breadcrumb

No "Step 14/31" counter — the trail names where you are:

Requirements ✓ · Design ✓ · Build ◐ Generate Code (GREEN) · Verify · Assess · Ship · Post-Ship
Requirements→ Design→ TA gate→ Build (TDD)→ Verify→ Assess→ TA gate→ Ship→ Post-Ship

Docs-only changes get their own route

A change touching nothing but documentation runs a 6-step docs workflow — issue → scope → draft → domain-routed review → reconcile → merge — instead of owing the full delivery trail. Mixed docs+code stays on the spine, so the short route can't be used to skip gates on real code. A merged change file counts as inactive, so a concluded change never satisfies the gate for the next one.
