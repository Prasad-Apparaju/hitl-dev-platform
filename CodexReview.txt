I reviewed the public repo at [Prasad-Apparaju/hitl-dev-platform](https://github.com/Prasad-Apparaju/hitl-dev-platform). The core thesis is strong: docs as executable team memory, AI as producer, humans as decision gates, traceability from issue to rollout. I would share it with a few former colleagues after cleanup, but I would not open it broadly yet as a reusable public process. There are credibility and runnability gaps that an outside reader will hit quickly.

**Main Gaps**

1. **The step count is inconsistent.**  
   The repo calls it a “22-step workflow” in several places, but the README says “24 steps,” the process overview lists 23, the main Claude template lists 28, and the feature guide also runs to 28. See [README step summary](https://github.com/Prasad-Apparaju/hitl-dev-platform/blob/main/README.md#L445-L456), [process overview](https://github.com/Prasad-Apparaju/hitl-dev-platform/blob/main/docs/playbook/process-overview.md#L19-L54), and [CLAUDE.md template](https://github.com/Prasad-Apparaju/hitl-dev-platform/blob/main/templates/CLAUDE.md.template#L23-L79).  
   This is the first thing I would fix. Pick one canonical workflow and make every doc point to it.

2. **Some docs contradict the TDD flow.**  
   The README’s workflow says tests come before code, but parts of the process docs and skill docs still show code before tests. That weakens one of your best ideas: “TDD as design.” Make the flow consistently: issue → impact → docs → test plan → tests RED → human test review → code GREEN → refactor → review.

3. **The repo still contains project-specific residue.**  
   Several public-facing docs mention PSR-Works, Prasad, Dilip, specific migration repos, and a “Showcase Roadmap.” Examples: [PM guide](https://github.com/Prasad-Apparaju/hitl-dev-platform/blob/main/docs/playbook/pm-guide.md#L1-L11), [adding-a-feature guide](https://github.com/Prasad-Apparaju/hitl-dev-platform/blob/main/docs/playbook/adding-a-feature.md#L1-L18), and [dev-practices skill](https://github.com/Prasad-Apparaju/hitl-dev-platform/blob/main/skills/dev-practices.md#L13-L68).  
   For colleagues, this is understandable. For the internet, it will look like a private migration playbook accidentally made public. Move PSR-Works material into `examples/psr-works/` or label it clearly as a case study.

4. **Quick-start references broken or missing files.**  
   The quick start tells users to copy `examples/greenfield/convention-checks.yaml`, but that file is not present. README/reference docs point to `templates/system-manifest.schema.yaml`, but the schema appears under `skills/generate-docs/templates/`. The generator README references `.github/workflows/manifest-check.yml`, but the repo has `ci/convention-check.yml`. These are small, but they matter because early adopters will follow the setup literally.

5. **License is inconsistent.**  
   `docs/quick-start.md` says MIT, but the GitHub repo has no detected license and I did not see a root `LICENSE` file. Before wider publication, add `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`, and a short support/status statement.

6. **The “light path” is acknowledged but not defined.**  
   The README admits the full workflow is too heavy for trivial changes and says there is “no clean answer.” That is honest, but for adoption you need a clear rule. Add a decision table like:
   - Trivial: issue or linked task, code, tests if applicable, review.
   - Bug fix: issue, regression test, fix, risk note, registry update if incident-related.
   - Normal feature: full workflow.
   - Cross-cutting/security/data migration: decomposition plan plus domain-scoped PRs.
   - P0 incident: fix first, docs within 48 hours.

7. **AI governance needs to be a first-class section.**  
   The process discusses correctness and traceability well, but public teams will ask: what can AI read, what data is prohibited, how are secrets protected, how do we handle proprietary code, model retention, generated code licensing, prompt injection, and supply-chain risk? You have security-related templates and patterns, but they are not central to the adoption workflow. Add an “AI Use and Data Handling Policy” section.

8. **Several claims need evidence or softer wording.**  
   Claims like “minutes rather than weeks,” “days, not sprints,” “one-week baseline,” “70% accurate,” and “debugging takes 10x longer” are compelling, but unsupported. For colleagues, that is fine as a working thesis. For public release, either label them as observed heuristics from your own projects or add a short case-study section with scope, team size, codebase size, elapsed time, and what worked/failed.

9. **The architect/lead bottleneck needs treatment.**  
   The process depends heavily on a lead/architect for design approval, integration verification, manifest accuracy, and ROI checks. That is workable for a small team, but the doc should state capacity limits and delegation rules. Add a RACI-style table and guidance for “what happens when the architect is unavailable.”

10. **The “docs update to match code” rule needs a guardrail.**  
   The README says when code diverges from docs, update docs to match. That can be right, but only after a human accepts that the implementation revealed a better design. Otherwise it normalizes drift. Rephrase to: “If implementation reveals a design gap, pause, decide whether the design changes, then either update docs or fix code.”

**What I’d Change Before Sharing With Former Colleagues**

Fix the step count, remove/genericize PSR-Works-specific references from the main path, add the missing license, repair broken setup paths, and add a short “how to read this” section that says this is an opinionated Claude Code implementation of a portable process.

**What I’d Change Before Opening It To The Internet**

Add the light-path decision table, AI governance/data-handling policy, contribution/security docs, link checker or CI validation for docs, a canonical workflow diagram, and one concrete case study. Also add a clear status banner: “early process draft,” “used on X type of project,” or “ready for pilot adoption.”

**Suggested Positioning**

Do not pitch it as “the new SDLC.” Pitch it as:

> An opinionated, docs-first HITL process for small teams using AI coding assistants, optimized for traceability, convention control, and safe acceleration.

That framing is defensible and matches what the repo actually contains.
